# SaunaFlow PoC 要件定義書 v0.1

## 1. プロジェクト概要

SaunaFlow は、サウナ室内の温熱環境を CFD（数値流体力学）シミュレーションで再現・評価するための Python ハーネスツールである。

### 目的
- サウナ室内の温度分布・湿度・気流を定量的に評価する
- 実験データとの比較検証により、シミュレーションの信頼性を確保する
- YAML ベースの宣言的ケース定義により、パラメータスタディを容易にする

### 技術スタック
| 要素 | 選定 |
|------|------|
| 計算エンジン | OpenFOAM (buoyantSimpleFoam / buoyantPimpleFoam) |
| ハーネス | Python 3.11+ |
| ケース定義 | YAML + JSON Schema バリデーション |
| 実行環境 | WSL2 (Ubuntu) 上の OpenFOAM |
| 開発環境 | Windows 11 + venv |

## 2. アーキテクチャ

### 基本原則
- **ハーネスがボス** — OpenFOAM は計算エンジンに過ぎない
- すべてのケース設定は宣言的（YAML/JSON）
- 同一入力からの決定論的・再現可能な実行
- Git が全入力を管理、結果は導出アーティファクト
- ソルバーは交換可能 — ハーネス API は不変

### モジュール構成
```
src/harness/
├── cli.py           — CLI エントリポイント (click)
├── schema.py        — YAML 読み込み + JSON Schema バリデーション
├── case_builder.py  — YAML → OpenFOAM ディレクトリ構造変換
├── mesh_runner.py   — メッシュ生成実行
├── solver_runner.py — ソルバー実行制御
├── probe_parser.py  — プローブ出力パース
├── kpi.py           — KPI 計算
├── validation.py    — CFD vs 実験データ比較
└── reporting.py     — レポート生成（プロット・表・合否判定）
```

### パイプライン
```
YAML定義 → schema検証 → case_builder → mesh_runner → solver_runner
    → probe_parser → kpi計算 → validation（実験データ比較） → reporting
```

## 3. KPI 定義

| ID | 名称 | 説明 |
|----|------|------|
| K-01 | 定常温度差 | 上段/下段ベンチの定常温度差分 |
| K-02 | ロウリュ後ピーク温度 | ロウリュ投入後の最高温度 |
| K-03 | ロウリュ後ピーク湿度 | ロウリュ投入後の最高湿度 |
| K-04 | ピーク到達時間 | ロウリュ後ピークに達するまでの時間 |
| K-05 | アウフグース風速ピーク | 顔面レベルの最大風速 |
| K-06 | 簡易温熱ストレス指数 | 温度・湿度・風速の複合指標 |
| K-07 | 上下相対差 | 上段/下段の相対温度差 |

## 4. バリデーション許容誤差（PoC 初期）

| 項目 | 許容範囲 |
|------|----------|
| 定常温度 | 主要プローブ点で ±3-5°C |
| ロウリュ後ピーク到達時間 | ±5-10 秒 |
| 風速ピーク | オーダー一致 + 方向一致 |
| 上下相対差 | トレンド一致 |

## 5. メッシュレベル

| レベル | セル数 | 用途 |
|--------|--------|------|
| M0 | 2-10 万 | 2D 粗メッシュ（初期検証） |
| M1 | 10-50 万 | 簡略 3D |
| M2 | 50-150 万 | PoC 本番 |
| M3 | 150-300 万 | 高精細（必要時のみ） |

## 6. フェーズ計画

### Phase 0: プロジェクト基盤 (現在)
- リポジトリ初期化、ディレクトリ構造
- YAML スキーマ定義、CLI スケルトン
- テスト基盤構築
- **完了基準**: `pytest tests/ -x -q` pass、`ruff check src/` pass、CLI help 表示

### Phase 1: ドライサウナ定常状態
- 簡易ジオメトリ定義（直方体）
- ヒーター境界条件
- プローブ配置と温度読み取り
- 収束判定
- **完了基準**: 温度成層トレンドの再現（K-01, K-07）

### Phase 2: ロウリュ導入
- 非定常計算への切り替え
- 水蒸気ソースの追加
- ピーク応答の計測
- **完了基準**: ピーク温度・湿度・到達時間の再現（K-02, K-03, K-04）

### Phase 3: アウフグース導入
- ジェット/運動量ソースの追加
- 局所風速・熱の評価
- **完了基準**: 風速差異の可視化（K-05, K-06）

### Phase 4: 実験データ検証
- センサーセットアップ定義
- CSV インポート・タイムスタンプ整合
- CFD vs 実験比較
- **完了基準**: 許容誤差内での比較成功

### Phase 5: 比較自動化
- バッチ比較実行
- 自動レポート生成
- **完了基準**: YAML ケース切替 → 再実行 → 比較 の自動化

## 7. 成功基準（全体）

1. 全テスト通過 (`pytest tests/ -x -q` が exit 0)
2. ドライサウナ定常状態で温度成層トレンドを再現
3. ロウリュ後の温度・湿度ピークタイミングが実験トレンドに一致
4. アウフグース条件間の風速差異が可視化可能
5. YAML ケース切替 → 再実行で同等の結果を再現

## 8. 開発ルール

- **テストファースト**: 新機能は必ずテストを先に書く
- **テストオラクル**: コミット前に必ず `pytest tests/ -x -q` を実行
- **Git worktree**: チケット作業は worktree + feature ブランチで独立
- **コード品質**: ruff, mypy, bandit による自動チェック
- **ドキュメント**: CHANGELOG.md に変更理由と失敗アプローチを記録
